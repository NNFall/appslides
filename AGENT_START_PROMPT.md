# Agent Start Prompt

## Готовый первый запрос для следующего агента

Скопируй текст ниже и используй его как первый большой запрос для нового агента.

```text
Ты работаешь в корне проекта, который уже является готовым каркасом мобильного AI-приложения.

У тебя есть два источника истины:

1. Текущий проект в этой папке.
Это не просто пример, а рабочий платформенный каркас:
- Flutter-клиент с единым chat UX;
- Python backend;
- YooKassa billing;
- локальное сохранение истории чата и файлов;
- отдельный Telegram admin bot;
- deploy-процесс и документация.

2. Отдельная папка другого Telegram-бота, которую я приложил.
Именно эта папка является источником предметной логики под новую тематику.
Из нее нужно брать:
- тексты;
- сценарии;
- prompts;
- продуктовые сущности;
- логику генерации;
- help/onboarding;
- admin-события;
- специфические правила продукта.

Твоя задача:
не переписывать платформу с нуля, а адаптировать текущий каркас под новую тематику.

Что нужно сохранить:
- единый чатовый интерфейс приложения;
- общую архитектуру Flutter-клиента;
- локальное сохранение истории чата;
- локальное сохранение и открытие файлов;
- client_id как ключ клиента/установки;
- backend-каркас;
- billing-каркас через YooKassa;
- admin-бота и admin-уведомления;
- deploy и GitHub workflow.

Что нужно заменить или адаптировать под новую тематику:
- стартовые тексты;
- help и onboarding;
- кнопки и шаги chat flow;
- prompts;
- продуктовые backend-сервисы;
- тип и формат результатов;
- продуктовые статусы;
- branding и пользовательские формулировки;
- тематические admin-уведомления.

Критически важно:
- не ломать transcript persistence;
- не ломать billing flow;
- не переносить платежную логику в Flutter;
- не удалять отдельный Telegram admin bot;
- не ломать deploy-процесс;
- не строить новый UI с нуля без необходимости.

Перед началом обязательно изучи эти документы в текущем проекте:
- PRODUCT_ADAPTATION_GUIDE.md
- IMPLEMENTATION_RULES.md
- AGENT_HANDOFF.md
- OPERATIONS.md
- APPSLIDES_PLAN.md
- README.md
- app/README.md
- backend/README.md

Также изучи ключевые файлы текущего каркаса:

Flutter:
- app/lib/features/chat/chat_screen.dart
- app/lib/features/billing/billing_controller.dart
- app/lib/data/api/appslides_api_client.dart
- app/lib/data/repositories/chat_transcript_repository.dart
- app/lib/data/repositories/saved_files_repository.dart
- app/lib/app/app_scope.dart

Backend:
- backend/src/api/presentations.py
- backend/src/domain/presentation_outline_service.py
- backend/src/domain/presentation_render_service.py
- backend/src/domain/presentation_prompts.py
- backend/src/integrations/text_generation.py
- backend/src/domain/billing_service.py
- backend/src/integrations/admin_notifier.py

Admin:
- telegram_admin_bot/handlers/admin.py

Как действовать:

Этап 1. Анализ
- Изучи новый Telegram-бот как источник предметной логики.
- Составь карту:
  - что в текущем проекте остается неизменным;
  - что заменяется под новую тематику;
  - какие сущности нового бота соответствуют текущим presentation/outline/template/artifact flow.

Этап 2. План
- Составь подробный пошаговый план миграции.
- План должен быть разбит на этапы:
  - Flutter/UI/chat flow;
  - backend/domain/API;
  - billing/admin/events;
  - branding/docs;
  - проверки и deploy.

Этап 3. Реализация
- Сначала адаптируй клиентский chat flow.
- Потом адаптируй backend product logic.
- Потом адаптируй admin-уведомления и статистику.
- Потом обнови документацию.

Этап 4. Проверка
- Прогони релевантные локальные проверки.
- Если менялся backend, после проверки подготовь deploy.
- Зафиксируй риски, открытые вопросы и что еще не доведено.

Формат работы:
- не останавливайся только на анализе;
- если изменений достаточно, вноси их;
- после крупных изменений обновляй документацию;
- после важных изменений коммить и пушь;
- если меняется backend, перевыкатывай сервер по правилам из OPERATIONS.md.

Что нужно выдать в самом начале ответа:

1. Краткое понимание задачи.
2. Таблицу или список:
   - что сохраняем;
   - что меняем;
   - какие файлы нового Telegram-бота критичны для анализа.
3. Подробный план работ по этапам.

Главный принцип:
нужно заменить смысл продукта, а не переписать платформу.
```

## Когда использовать этот промпт

Подходит для сценария, когда:

- ты копируешь текущий `appslides` в новую папку;
- прикладываешь другую папку Telegram-бота;
- хочешь, чтобы новый агент использовал `appslides` как базу, а не строил все заново.

## Что можно дописать вручную перед отправкой

Обычно полезно добавить в начало или конец промпта:

- название новой тематики;
- какие именно результаты должен выдавать продукт;
- какие экраны или сценарии особенно важны;
- нужно ли сохранять совместимость со старыми API-именами;
- нужно ли сразу готовить Android APK, backend deploy и admin-бота.
